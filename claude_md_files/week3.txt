================================================================================
DAY 1-2: REDIS FUNDAMENTALS
================================================================================

BEFORE YOU TOUCH ANY COMMANDS - READ AND UNDERSTAND THIS:
----------------------------------------------------------

>> WHAT IS REDIS? <<

Redis = Remote Dictionary Server. It's an IN-MEMORY data store.

Think about your current TinyScale flow when someone clicks a short URL:

    User clicks short URL
        ↓
    Nginx → App → Query Postgres (disk I/O) → Return redirect
        ↓
    ~5-20ms for the DB query alone

Now imagine if the short_code → original_url mapping was already in memory:

    User clicks short URL
        ↓
    Nginx → App → Check Redis (in-memory) → Return redirect
        ↓
    ~0.1-0.5ms for the Redis lookup

That's 10-100x faster. That's why Redis exists.

>> BUT WHY NOT JUST USE A JavaScript OBJECT/MAP? <<

Good question. You COULD cache in a JS Map:

    const cache = new Map();
    cache.set("Ax7bK2", "https://google.com");

Problems:
1. MEMORY LIMIT: Your Node.js process has limited heap memory (~1.5GB default).
   Redis can use as much RAM as your server has.

2. PROCESS RESTART: When your app restarts, the Map is gone.
   Redis persists data (optional) and survives app restarts.

3. MULTIPLE APP INSTANCES: In Week 7, you'll run 3 replicas of your app.
   Each has its own Map — they can't share cache.
   Redis is a SEPARATE service — all app instances share it.

4. DATA STRUCTURES: Redis has sorted sets, lists, hashes, streams, pub/sub.
   A JS Map is just key-value. Redis is a data structure server.

5. TTL (TIME TO LIVE): Redis can auto-expire keys. With a Map, you'd have
   to implement your own expiration logic.

>> WHY IS REDIS FAST IF IT'S JUST ANOTHER SERVER? <<

Redis is fast because:
1. IN-MEMORY: All data lives in RAM. No disk I/O for reads.
   RAM access: ~100 nanoseconds. Disk access: ~10 milliseconds. That's 100,000x faster.

2. SINGLE-THREADED: No locking, no context switching, no race conditions.
   "Wait, single-threaded = slow?" — No. Redis is I/O-bound, not CPU-bound.
   It uses an event loop (like Node.js!) to handle thousands of connections.

3. EFFICIENT PROTOCOLS: Redis uses a simple text protocol (RESP).
   Minimal parsing overhead compared to SQL.

4. OPTIMIZED DATA STRUCTURES: Redis's internal implementations
   (skip lists, hash tables, ziplist) are highly optimized for memory and speed.

>> HOW IS REDIS DIFFERENT FROM POSTGRES? <<

    Postgres                            Redis
    ───────                             ─────
    Stores on DISK (durable)            Stores in MEMORY (fast)
    Complex queries (JOIN, GROUP BY)    Simple key-value operations
    ACID transactions                   Atomic single-command operations
    Best for: source of truth           Best for: cache, sessions, counters
    Handles 1K-10K queries/sec          Handles 100K+ operations/sec
    Data survives power loss            Data LOST on restart (unless configured)

RULE: Postgres is your source of truth. Redis is your speed layer.
If Redis dies, your app should still work — just slower.

STEP 1: ADD REDIS TO DOCKER-COMPOSE
================================================================================

Open your docker-compose.yml and add the Redis service.

Add this block to the `services:` section (after the db service):

  # ---- REDIS ----
  redis:
    image: redis:7-alpine
    # Redis 7 on Alpine Linux — small and fast
    ports:
      - "6379:6379"
      # 6379 is Redis's default port
      # Mapping to host so you can use redis-cli from your machine
    volumes:
      - redisdata:/data
      # Redis can persist data to disk (RDB snapshots / AOF log)
      # /data is where Redis saves its dump.rdb file
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 5s
      retries: 5
      # redis-cli ping should return PONG if Redis is healthy
    networks:
      - tinyscale-network
    command: redis-server --maxmemory 128mb --maxmemory-policy allkeys-lru
    # --maxmemory 128mb = limit Redis to 128MB of RAM
    #   In production, you'd set this higher based on your server.
    #   For local dev, 128MB is plenty.
    # --maxmemory-policy allkeys-lru = when Redis is full, evict Least Recently Used keys
    #   Other policies:
    #     volatile-lru = evict only keys WITH an expire set
    #     allkeys-random = evict random keys
    #     noeviction = return errors when memory is full (dangerous!)
    #   LRU is the safest default for a cache.

Also add `redisdata` to the volumes section at the bottom:

volumes:
  pgdata:
  redisdata:
    # Persists Redis data between container restarts
    # Not strictly necessary for a cache (it's expendable)
    # but useful so you don't lose data during docker-compose restart

Now update your `app` service to depend on Redis:

  app:
    # ... existing config ...
    depends_on:
      db:
        condition: service_healthy
      migrate:
        condition: service_completed_successfully
      redis:
        condition: service_healthy
    environment:
      DATABASE_URL: postgres://dev:dev@db:5432/tinyscale
      JWT_SECRET: docker-dev-secret-change-in-production
      REDIS_URL: redis://redis:6379
      PORT: 3000
      HOST: 0.0.0.0
      NODE_ENV: production

KEY THINGS TO UNDERSTAND:

1. `redis://redis:6379` — The hostname `redis` is the service name in docker-compose.
   Docker DNS resolves it to the Redis container's IP, just like `db` for Postgres.

2. `--maxmemory-policy allkeys-lru` — This is CRITICAL for a cache.
   Without it, Redis fills up and returns errors. With LRU, old cached data
   is automatically evicted to make room for new data.

3. Redis image is already configured for production use. Unlike Postgres,
   there's no user/password by default. In production, you'd add:
     command: redis-server --requirepass yourpassword
   For local dev, no password is fine.

STEP 2: START REDIS AND EXPLORE WITH redis-cli
================================================================================

Bring everything up:

    docker compose up -d redis

Verify Redis is running:

    docker compose ps

You should see the redis container as "healthy".

Now connect with redis-cli:

    docker compose exec redis redis-cli

You're now inside the Redis CLI. Let's explore every data structure you'll use.


>> STRINGS (the basic key-value) <<

    SET greeting "hello world"
    GET greeting
    # Returns: "hello world"

    SET counter 0
    INCR counter
    INCR counter
    INCR counter
    GET counter
    # Returns: "3"
    # INCR is ATOMIC — safe even with concurrent access

    SET mykey "expires soon" EX 10
    # EX 10 = expires in 10 seconds
    TTL mykey
    # Returns: seconds remaining (e.g., 8)
    # Wait 10 seconds...
    GET mykey
    # Returns: (nil) — it's gone!

    # This is how we'll cache URL lookups:
    SET "url:Ax7bK2" "https://google.com" EX 3600
    # Cache short code → URL for 1 hour (3600 seconds)
    GET "url:Ax7bK2"
    # Returns: "https://google.com"

WHY THIS MATTERS FOR TINYSCALE:
    Every redirect does: SELECT * FROM urls WHERE short_code = 'Ax7bK2'
    With Redis: GET "url:Ax7bK2" — 100x faster, zero DB load.


>> HASHES (like a JavaScript object) <<

    HSET user:123 name "Aslam" email "aslam@test.com" role "admin"
    HGET user:123 name
    # Returns: "Aslam"

    HGETALL user:123
    # Returns: name "Aslam" email "aslam@test.com" role "admin"

    HINCRBY user:123 loginCount 1
    # Increments the loginCount field by 1

WHY THIS MATTERS:
    Sessions, user profiles, caching objects with multiple fields.
    Instead of serializing/deserializing JSON, access individual fields.


>> LISTS (ordered, like an array) <<

    LPUSH queue:clicks "click-event-1"
    LPUSH queue:clicks "click-event-2"
    LPUSH queue:clicks "click-event-3"
    # LPUSH = push to LEFT (front) of list

    LRANGE queue:clicks 0 -1
    # Returns all elements (0 to -1 means start to end)

    RPOP queue:clicks
    # Returns and removes from RIGHT (end) — "click-event-1"
    # This is FIFO queue behavior: LPUSH + RPOP

WHY THIS MATTERS:
    BullMQ uses Redis lists (and streams) as its job queue.
    Jobs are pushed in, workers pop them out.


>> SETS (unique values, no duplicates) <<

    SADD active:users "user-1" "user-2" "user-3" "user-1"
    SMEMBERS active:users
    # Returns: "user-1" "user-2" "user-3" (no duplicate user-1)

    SISMEMBER active:users "user-2"
    # Returns: 1 (true) — O(1) lookup!

    SCARD active:users
    # Returns: 3 (count of members)

WHY THIS MATTERS:
    Tracking unique visitors, deduplication, feature flags.


>> SORTED SETS (ranked data) <<

    ZADD leaderboard 100 "url-abc" 250 "url-def" 50 "url-ghi"
    # ZADD key score member — each member has a score

    ZRANGE leaderboard 0 -1 WITHSCORES
    # Returns sorted by score (ascending): ghi(50), abc(100), def(250)

    ZREVRANGE leaderboard 0 2 WITHSCORES
    # Returns top 3 by score (descending): def(250), abc(100), ghi(50)

    ZINCRBY leaderboard 1 "url-abc"
    # Increment url-abc's score by 1 (now 101)

WHY THIS MATTERS:
    Real-time click counters! Every click: ZINCRBY leaderboard 1 "url-abc"
    Get top 10 most clicked URLs: ZREVRANGE leaderboard 0 9 WITHSCORES
    No database query needed. Updated in real-time.


>> KEY EXPIRATION AND TTL <<

    SET temp "I disappear" EX 30
    TTL temp
    # Returns: 30 (seconds remaining)

    PERSIST temp
    # Removes expiration — key lives forever now

    EXPIRE temp 60
    # Set expiration on existing key — 60 seconds

    PTTL temp
    # TTL in milliseconds (more precise)

WHY THIS MATTERS:
    Cache invalidation! Set TTL on cached URL lookups.
    After TTL expires, next request hits Postgres and re-caches.
    This prevents stale data from living forever.


>> USEFUL COMMANDS <<

    KEYS *
    # List ALL keys (DON'T use in production — scans entire DB)
    # Use SCAN instead for production

    DBSIZE
    # How many keys exist

    INFO memory
    # How much memory Redis is using

    FLUSHALL
    # Delete EVERYTHING (careful!)

    TYPE mykey
    # What data type is this key? (string, list, set, hash, zset)

    EXIT
    # Leave redis-cli

>> WHY ioredis AND NOT redis (the official client)? <<

    ioredis:
    - Better TypeScript support
    - Built-in cluster support
    - Built-in Sentinel support (high availability)
    - Automatic reconnection
    - Pipeline and transaction support
    - Used by BullMQ (which we'll add in Day 5-6)

    The official `redis` package (v4+) is also good, but ioredis is
    the de facto standard in the Node.js ecosystem.

STEP 5: CREATE THE REDIS PLUGIN
================================================================================

Create a new file: src/plugins/redis.plugin.ts

--- START OF src/plugins/redis.plugin.ts ---

import fp from "fastify-plugin";
import { type FastifyInstance } from "fastify";
import Redis from "ioredis";
import { env } from "../config/env.js";

// Extend Fastify's type system so app.redis is available everywhere
declare module "fastify" {
  interface FastifyInstance {
    redis: Redis;
  }
}

export const redisPlugin = fp(async (fastify: FastifyInstance) => {
  const redis = new Redis(env.REDIS_URL, {
    // Connection settings
    maxRetriesPerRequest: 3,
    // If Redis is unreachable, retry 3 times before giving up on that request.
    // BullMQ requires this to be null (we'll handle that separately in Day 5-6).

    retryStrategy(times) {
      // How long to wait before reconnecting after disconnect
      const delay = Math.min(times * 200, 5000);
      // First retry: 200ms, second: 400ms, ... max: 5000ms
      fastify.log.warn(`Redis reconnecting... attempt ${times}, delay ${delay}ms`);
      return delay;
    },

    // When Redis connects/disconnects, log it
    lazyConnect: false,
    // lazyConnect: false = connect immediately when created
    // lazyConnect: true = connect on first command (useful for optional Redis)
  });

  // Log connection events
  redis.on("connect", () => {
    fastify.log.info("Redis connected");
  });

  redis.on("error", (err) => {
    fastify.log.error(err, "Redis connection error");
    // DON'T throw — let the app continue without Redis (graceful degradation)
  });

  redis.on("close", () => {
    fastify.log.warn("Redis connection closed");
  });

  // Decorate Fastify with the Redis client
  fastify.decorate("redis", redis);

  // Clean up on app shutdown
  fastify.addHook("onClose", async () => {
    fastify.log.info("Closing Redis connection");
    await redis.quit();
    // redis.quit() sends QUIT command — graceful disconnect
    // redis.disconnect() is forceful — don't use unless necessary
  });
});

--- END OF src/plugins/redis.plugin.ts ---


>> WHAT JUST HAPPENED? <<

1. We created a Fastify plugin (just like db.plugin.ts and auth.plugin.ts)
2. It creates an ioredis client connected to REDIS_URL
3. It decorates the Fastify instance so you can use `app.redis` anywhere
4. It logs connection events (connect, error, close)
5. It gracefully disconnects when the app shuts down
6. It retries connection with exponential backoff if Redis goes down

The pattern is identical to your db.plugin.ts — same approach, different service.

================================================================================
STEP 1: UNDERSTAND THE CACHE-ASIDE PATTERN
================================================================================

This is the most common caching pattern. Here's how it works:

    WITHOUT CACHE (current):
    ┌─────────┐     ┌──────┐     ┌──────────┐
    │  Client  │────→│  App │────→│ Postgres │
    │          │←────│      │←────│          │
    └─────────┘     └──────┘     └──────────┘
    Every request hits the database. Every. Single. One.

    WITH CACHE-ASIDE:
    ┌─────────┐     ┌──────┐     ┌───────┐
    │  Client  │────→│  App │────→│ Redis │ ← Check cache FIRST
    │          │←────│      │←────│       │
    └─────────┘     └──────┘     └───────┘
                        │              ↑
                        │ CACHE MISS   │ Store result
                        ↓              │
                    ┌──────────┐       │
                    │ Postgres │───────┘
                    │          │
                    └──────────┘

    The flow:
    1. App receives request for short_code "Ax7bK2"
    2. App checks Redis: GET "url:Ax7bK2"
    3. CACHE HIT → Redis returns the URL → respond immediately (fast!)
    4. CACHE MISS → Redis returns nil → query Postgres → store result in Redis → respond
    5. Next request for "Ax7bK2" hits the cache (step 3)

    This is called "cache-aside" (or "lazy loading") because:
    - The cache is populated ON DEMAND (not pre-loaded)
    - The app manages the cache (not the database)
    - The cache sits ASIDE the main data path

>> OTHER CACHING PATTERNS (know they exist): <<

    Write-Through: App writes to cache AND database on every write.
        Pro: Cache is always up-to-date. Con: Writes are slower.

    Write-Behind: App writes to cache, cache async writes to database.
        Pro: Writes are fast. Con: Data loss risk if cache crashes before writing to DB.

    Read-Through: Cache itself fetches from database on miss (cache is smart).
        Pro: Simpler app code. Con: Cache needs DB access.

    We use Cache-Aside because it's the simplest, most predictable,
    and gives us full control. This is what 90% of applications use.


================================================================================
STEP 2: CREATE A CACHE SERVICE
================================================================================

Create a new file: src/services/cache.service.ts

--- START OF src/services/cache.service.ts ---

import { type FastifyInstance } from "fastify";

// Default TTL: 1 hour (in seconds)
const DEFAULT_TTL = 3600;

// Prefix all cache keys to avoid collisions with other Redis uses
// (BullMQ, sessions, rate limits will also use Redis)
const KEY_PREFIX = "cache:";

/**
 * Cache-aside: Get from cache, or fetch from source and cache the result.
 *
 * @param app - Fastify instance (for app.redis)
 * @param key - Cache key (will be prefixed with "cache:")
 * @param fetchFn - Function to call on cache miss (fetches from DB)
 * @param ttl - Time to live in seconds (default: 3600 = 1 hour)
 * @returns The cached or freshly fetched value
 */
export async function cacheGet<T>(
  app: FastifyInstance,
  key: string,
  fetchFn: () => Promise<T>,
  ttl: number = DEFAULT_TTL
): Promise<T> {
  const cacheKey = `${KEY_PREFIX}${key}`;

  try {
    // Step 1: Try to get from cache
    const cached = await app.redis.get(cacheKey);

    if (cached !== null) {
      // CACHE HIT — parse and return
      app.log.debug({ key: cacheKey }, "Cache HIT");
      return JSON.parse(cached) as T;
    }
  } catch (err) {
    // If Redis fails, log and continue to database
    // This is graceful degradation — app works without cache
    app.log.warn(err, "Redis GET failed, falling through to database");
  }

  // Step 2: CACHE MISS — fetch from the source (database)
  app.log.debug({ key: cacheKey }, "Cache MISS");
  const value = await fetchFn();

  // Step 3: Store in cache for next time
  try {
    await app.redis.set(cacheKey, JSON.stringify(value), "EX", ttl);
    // "EX" = expire in seconds
    // After `ttl` seconds, Redis automatically deletes this key
  } catch (err) {
    // Cache write failure is non-fatal — next request will try again
    app.log.warn(err, "Redis SET failed");
  }

  return value;
}

/**
 * Invalidate (delete) a cached key.
 * Call this when the underlying data changes (e.g., URL is updated or deleted).
 */
export async function cacheInvalidate(
  app: FastifyInstance,
  key: string
): Promise<void> {
  const cacheKey = `${KEY_PREFIX}${key}`;

  try {
    await app.redis.del(cacheKey);
    app.log.debug({ key: cacheKey }, "Cache invalidated");
  } catch (err) {
    app.log.warn(err, "Redis DEL failed");
  }
}

/**
 * Invalidate all keys matching a pattern.
 * Use sparingly — SCAN-based deletion is slower than direct DEL.
 *
 * Example: cacheInvalidatePattern(app, "stats:*") — clears all stats caches
 */
export async function cacheInvalidatePattern(
  app: FastifyInstance,
  pattern: string
): Promise<void> {
  const fullPattern = `${KEY_PREFIX}${pattern}`;

  try {
    // SCAN is safe for production (unlike KEYS which blocks Redis)
    let cursor = "0";
    do {
      const [nextCursor, keys] = await app.redis.scan(
        cursor,
        "MATCH",
        fullPattern,
        "COUNT",
        100
      );
      cursor = nextCursor;

      if (keys.length > 0) {
        await app.redis.del(...keys);
        app.log.debug({ pattern: fullPattern, deleted: keys.length }, "Cache pattern invalidated");
      }
    } while (cursor !== "0");
  } catch (err) {
    app.log.warn(err, "Redis pattern invalidation failed");
  }
}

--- END OF src/services/cache.service.ts ---


>> WHAT JUST HAPPENED? <<

1. `cacheGet<T>()` — The main function. Implements cache-aside pattern.
   Generic type `<T>` means it works with any data type.

2. `cacheInvalidate()` — Deletes a specific cache key.
   You call this when data changes (URL updated, deleted, etc.).

3. `cacheInvalidatePattern()` — Deletes all keys matching a pattern.
   Uses SCAN (not KEYS) because KEYS blocks Redis with large datasets.

4. EVERY Redis call is wrapped in try/catch. If Redis fails, the app
   continues normally (just slower, hitting Postgres directly).
   This is CRITICAL — your cache should never break your app.

5. All keys are prefixed with "cache:" to avoid collisions with
   BullMQ keys, session keys, rate limit keys, etc.

  // Use cache-aside pattern: check Redis first, then Postgres
  const url = await cacheGet(
    app,
    `url:${code}`,    // Cache key: "cache:url:Ax7bK2"
    async () => {
      // This only runs on CACHE MISS
      const [result] = await app.db
        .select()
        .from(urls)
        .where(eq(urls.shortCode, code))
        .limit(1);

      return result ?? null;
    },
    3600   // TTL: 1 hour. After this, next request re-fetches from Postgres.
  );

  if (!url) {
    throw new NotFoundError("Short URL not found");
  }

  // Check expiration
  if (url.expiresAt && new Date(url.expiresAt) < new Date()) {
    // URL expired — invalidate the cache so we don't keep serving it
    await cacheInvalidate(app, `url:${code}`);
    throw new NotFoundError("Short URL has expired");
  }

  return url;
}

--- END OF MODIFICATION ---


>> WHAT CHANGED? <<

BEFORE:
    resolveShortCode → Postgres query → return

AFTER:
    resolveShortCode → Redis GET → HIT? return immediately
                                  → MISS? Postgres query → Redis SET → return

The first request for "Ax7bK2" hits Postgres (miss). Every subsequent request
for the next hour hits Redis (hit). For a popular short URL that gets clicked
1000 times/day, that's 999 Redis hits and 1 Postgres query instead of 1000.


>> WHAT ABOUT CACHE INVALIDATION? <<

"There are only two hard things in Computer Science: cache invalidation
and naming things." — Phil Karlton

When do we need to invalidate (delete) the cache?

1. URL expires → we handle this above (invalidate on expiry check)
2. URL is deleted → we'd add cacheInvalidate when implementing delete
3. URL is updated → we'd add cacheInvalidate when implementing update

For now, our URLs are immutable (can't be updated or deleted via API),
so TTL-based expiration is sufficient. The cache auto-expires after 1 hour,
and the next request re-fetches from Postgres.

If you later add a DELETE /api/urls/:code endpoint, you MUST also call:
    await cacheInvalidate(app, `url:${code}`);
Otherwise users will still be redirected to a deleted URL for up to 1 hour.

>> THE PROBLEM WE'RE SOLVING <<

Look at your current redirect handler (src/modules/url/url.handler.ts):

    export async function redirectHandler(...) {
      const url = await resolveShortCode(request.server, request.params.code);

      // Fire-and-forget click recording
      recordClick(request.server, url.id, url.shortCode, {
        referrer: ...,
        userAgent: ...,
        ip: ...,
      }).catch((err) => request.log.error(err, "Failed to record click"));

      return reply.redirect(url.originalUrl);
    }

The redirect is fast because we "fire-and-forget" the click recording.
But there are problems:

1. FIRE-AND-FORGET IS UNRELIABLE:
   If the Postgres INSERT fails, we just log it and lose the click data.
   No retry. No recovery. The analytics are permanently wrong.

2. NO BACKPRESSURE:
   If 10,000 users click simultaneously, we fire 10,000 concurrent Postgres
   INSERTs. The database connection pool (max: 20) becomes a bottleneck.
   Queries queue up, latency spikes, and some may timeout.

3. NO BATCHING:
   Each click = 1 INSERT query. 10,000 clicks = 10,000 queries.
   A single INSERT with 10,000 rows would be 100x more efficient.

4. MIXED WORKLOADS:
   The same app process handles redirects (user-facing, latency-critical)
   AND click recording (background, throughput-critical).
   A slow click recording can starve the event loop from handling redirects.

>> THE SOLUTION: JOB QUEUES <<

Instead of processing clicks inline, we QUEUE them for later processing:

    BEFORE:
    Request → Resolve URL → INSERT click → Redirect
                             ↑ slow, unreliable

    AFTER:
    Request → Resolve URL → PUSH to queue → Redirect (immediate!)
                             ↑ instant (~1ms)

    Meanwhile, in a separate process:
    Worker → POP from queue → Batch INSERT clicks → Confirm processed

Benefits:
1. FAST REDIRECT: Pushing to a queue is ~1ms. Redirect happens immediately.
2. RELIABLE: If the worker crashes, the job stays in the queue. It's retried.
3. BATCHING: Worker can collect 100 clicks and INSERT them in one query.
4. BACKPRESSURE: Worker processes at its own pace. Queue absorbs bursts.
5. SEPARATION: Redirect handler is isolated from click processing problems.

>> WHAT IS BULLMQ? <<

BullMQ is a Redis-based job queue for Node.js. It uses Redis lists and streams
to implement a reliable, distributed job queue with:

    - Job persistence (jobs survive app restarts — they're in Redis)
    - Automatic retries (failed jobs are retried with exponential backoff)
    - Dead letter queue (jobs that fail N times are moved aside, not lost)
    - Job prioritization (some jobs can be processed before others)
    - Concurrency control (process N jobs simultaneously)
    - Job scheduling (process jobs at a specific time)
    - Job progress (report progress during long-running jobs)

BullMQ stores everything in Redis. It uses Redis's atomic operations (BRPOPLPUSH,
Lua scripts) to ensure jobs are never lost and never processed twice.

>> WHY BULLMQ AND NOT JUST Redis LPUSH/RPOP? <<

You COULD build a simple queue with Redis lists:

    // Producer
    await redis.lpush("queue:clicks", JSON.stringify(clickData));

    // Consumer
    const data = await redis.brpop("queue:clicks", 0);
    await processClick(data);

Problems with this DIY approach:
1. No retry logic (if processing fails, the job is gone)
2. No dead letter queue (no way to inspect failed jobs)
3. No concurrency control (process one job at a time? multiple? how many?)
4. No monitoring (how many jobs pending? how many failed?)
5. No job deduplication
6. Race conditions with multiple consumers

BullMQ solves all of these. It's production-tested at scale.
Companies like Adobe, Microsoft, and Salesforce use it.

import { Queue } from "bullmq";
import { env } from "../config/env.js";

// Define the shape of a click job's data
export interface ClickJobData {
  urlId: string;
  shortCode: string;
  referrer: string | null;
  userAgent: string | null;
  ip: string | null;
  clickedAt: string;  // ISO timestamp — serialize dates as strings for JSON
}

// Queue name — must be the same in producer and consumer
export const CLICK_QUEUE_NAME = "clicks";

// Create the queue (producer side)
// This is the "mailbox" where you drop off jobs.
export function createClickQueue(): Queue<ClickJobData> {
  return new Queue<ClickJobData>(CLICK_QUEUE_NAME, {
    connection: {
      host: new URL(env.REDIS_URL).hostname,
      port: parseInt(new URL(env.REDIS_URL).port || "6379"),
      // BullMQ needs host/port, not a URL string
      maxRetriesPerRequest: null,
      // IMPORTANT: BullMQ REQUIRES this to be null.
      // Without it, you'll get: "maxRetriesPerRequest must be null"
    },
    defaultJobOptions: {
      attempts: 3,
      // If a job fails, retry up to 3 times

      backoff: {
        type: "exponential",
        delay: 1000,
        // Retry delays: 1s, 2s, 4s (exponential backoff)
        // This prevents hammering a failing service
      },

      removeOnComplete: {
        count: 1000,
        // Keep the last 1000 completed jobs for debugging/monitoring
        // Older completed jobs are automatically deleted from Redis
        // Without this, completed jobs accumulate and waste Redis memory
      },

      removeOnFail: {
        count: 5000,
        // Keep the last 5000 failed jobs for investigation
        // You need to see WHY jobs failed
      },
    },
  });
}

>> WHAT JUST HAPPENED? <<

We defined:
1. `ClickJobData` — the TypeScript interface for click event data
2. `CLICK_QUEUE_NAME` — "clicks" — the queue name (shared between producer and consumer)

--- START OF src/jobs/click-worker.ts ---

import { Worker, type Job } from "bullmq";
import { env } from "../config/env.js";
import { CLICK_QUEUE_NAME, type ClickJobData } from "./click-queue.js";
import { drizzle } from "drizzle-orm/node-postgres";
import pg from "pg";
import { clicks } from "../db/schema/index.js";
import Redis from "ioredis";

// Batch settings
const BATCH_SIZE = 50;
const BATCH_TIMEOUT_MS = 5000; // 5 seconds

// Buffer to collect clicks before flushing to DB
let clickBuffer: ClickJobData[] = [];
let flushTimer: ReturnType<typeof setTimeout> | null = null;

// We need our own DB connection (not the app's)
// because the worker runs in a separate context
let pool: pg.Pool | null = null;
let db: ReturnType<typeof drizzle> | null = null;
let redis: Redis | null = null;

function getDb() {
  if (!db) {
    pool = new pg.Pool({
      connectionString: env.DATABASE_URL,
      max: 5, // Worker doesn't need many connections
    });
    db = drizzle(pool);
  }
  return db;
}

function getRedis() {
  if (!redis) {
    redis = new Redis(env.REDIS_URL);
  }
  return redis;
}

/**
 * Flush the click buffer to Postgres in a single batch INSERT.
 * This is dramatically more efficient than individual INSERTs.
 */
async function flushClickBuffer(): Promise<void> {
  if (clickBuffer.length === 0) return;

  // Grab current buffer and reset
  const batch = clickBuffer;
  clickBuffer = [];

  if (flushTimer) {
    clearTimeout(flushTimer);
    flushTimer = null;
  }

  const database = getDb();

  try {
    // BATCH INSERT — one query for N clicks instead of N queries
    await database.insert(clicks).values(
      batch.map((click) => ({
        urlId: click.urlId,
        referrer: click.referrer,
        userAgent: click.userAgent,
        ip: click.ip,
        clickedAt: new Date(click.clickedAt),
      }))
    );

    console.log(`[ClickWorker] Flushed ${batch.length} clicks to database`);

    // Update Redis click leaderboard for each unique shortCode in the batch
    const r = getRedis();
    const shortCodeCounts = new Map<string, number>();
    for (const click of batch) {
      shortCodeCounts.set(
        click.shortCode,
        (shortCodeCounts.get(click.shortCode) ?? 0) + 1
      );
    }
    for (const [code, count] of shortCodeCounts) {
      await r.zincrby("clicks:leaderboard", count, code);
      // Invalidate stats cache since clicks changed
      await r.del(`cache:stats:${code}`);
    }
  } catch (err) {
    console.error(`[ClickWorker] Flush failed for ${batch.length} clicks:`, err);
    // Put failed clicks back in the buffer for next flush
    // This is a simple retry — BullMQ also retries the individual jobs
    clickBuffer.unshift(...batch);
    throw err; // Re-throw so BullMQ marks the job as failed
  }
}

/**
 * Schedule a flush after BATCH_TIMEOUT_MS if one isn't already scheduled.
 * This ensures clicks are written even during low-traffic periods.
 */
function scheduleFlush(): void {
  if (!flushTimer) {
    flushTimer = setTimeout(() => {
      flushClickBuffer().catch((err) => {
        console.error("[ClickWorker] Scheduled flush failed:", err);
      });
    }, BATCH_TIMEOUT_MS);
  }
}

/**
 * Create and start the click worker.
 * The worker pulls jobs from the "clicks" queue and processes them.
 */
export function createClickWorker(): Worker<ClickJobData> {
  const worker = new Worker<ClickJobData>(
    CLICK_QUEUE_NAME,
    async (job: Job<ClickJobData>) => {
      // Add click to buffer
      clickBuffer.push(job.data);

      // Flush if buffer is full
      if (clickBuffer.length >= BATCH_SIZE) {
        await flushClickBuffer();
      } else {
        // Schedule a flush in case traffic is slow
        scheduleFlush();
      }
    },
    {
      connection: {
        host: new URL(env.REDIS_URL).hostname,
        port: parseInt(new URL(env.REDIS_URL).port || "6379"),
        maxRetriesPerRequest: null,
      },
      concurrency: 10,
      // Process up to 10 jobs concurrently.
      // This fills the buffer faster for more efficient batch inserts.
      // Each job just adds to the buffer (fast), so high concurrency is fine.

      limiter: {
        max: 1000,
        duration: 1000,
        // Process at most 1000 jobs per second.
        // This prevents the worker from overwhelming Postgres.
      },
    }
  );

  // Event handlers — important for monitoring and debugging
  worker.on("completed", (job) => {
    // Individual job completed — don't log each one (too noisy)
    // Batch flushes are logged in flushClickBuffer()
  });

  worker.on("failed", (job, err) => {
    console.error(
      `[ClickWorker] Job ${job?.id} failed (attempt ${job?.attemptsMade}/${job?.opts?.attempts}):`,
      err.message
    );
  });

  worker.on("error", (err) => {
    console.error("[ClickWorker] Worker error:", err);
  });

  worker.on("stalled", (jobId) => {
    console.warn(`[ClickWorker] Job ${jobId} stalled (took too long, will be retried)`);
    // A "stalled" job means the worker took too long to process it.
    // BullMQ assumes the worker crashed and re-queues the job.
    // Default stall interval: 30 seconds.
  });

  console.log("[ClickWorker] Worker started, waiting for jobs...");

  return worker;
}

/**
 * Gracefully shut down the worker.
 * Flushes any remaining clicks in the buffer before stopping.
 */
export async function shutdownClickWorker(worker: Worker): Promise<void> {
  console.log("[ClickWorker] Shutting down...");

  // Flush remaining clicks
  await flushClickBuffer();

  // Close the worker (stops pulling new jobs)
  await worker.close();

  // Close DB pool
  if (pool) {
    await pool.end();
  }

  // Close Redis
  if (redis) {
    await redis.quit();
  }

  console.log("[ClickWorker] Shutdown complete");
}

--- END OF src/jobs/click-worker.ts ---


>> WHAT DOES THIS WORKER DO? <<

1. PULLS jobs from the "clicks" queue in Redis
2. BUFFERS them in memory (collects clicks)
3. FLUSHES to Postgres in batches when:
   - Buffer reaches 50 clicks (BATCH_SIZE), OR
   - 5 seconds pass with no full batch (BATCH_TIMEOUT_MS)
4. Updates the Redis click leaderboard for each batch
5. Invalidates stats cache for affected URLs
6. RETRIES failed flushes (BullMQ automatic retry with exponential backoff)
7. Logs failed/stalled jobs for debugging

The key insight: instead of 1000 individual INSERTs, we do 20 batch INSERTs
of 50 rows each. This is dramatically faster because:
- Fewer round-trips to Postgres
- Fewer transaction commits
- Better disk I/O patterns (sequential vs random)

--- MODIFY src/server.ts ---

Your server.ts should look like this (add the worker imports and startup):

import { buildApp } from "./app.js";
import { env } from "./config/env.js";
import { createClickWorker, shutdownClickWorker } from "./jobs/click-worker.js";

async function main() {
  const app = await buildApp();

  // Start the click worker (processes click jobs from the queue)
  const clickWorker = createClickWorker();

  // Graceful shutdown
  const shutdown = async (signal: string) => {
    app.log.info(`Received ${signal}, shutting down gracefully...`);

    // Shut down worker first (flush remaining clicks)
    await shutdownClickWorker(clickWorker);

    // Then close the app (closes DB pool, Redis, etc.)
    await app.close();

    process.exit(0);
  };

  process.on("SIGTERM", () => shutdown("SIGTERM"));
  process.on("SIGINT", () => shutdown("SIGINT"));

  try {
    await app.listen({ port: env.PORT, host: env.HOST });
  } catch (err) {
    app.log.error(err);
    await shutdownClickWorker(clickWorker);
    process.exit(1);
  }
}

main();

--- END OF MODIFICATION ---


>> WHY GRACEFUL SHUTDOWN MATTERS: <<

Without graceful shutdown:
1. App gets SIGTERM (e.g., Docker stopping the container)
2. Process dies immediately
3. Clicks in the worker's buffer are LOST (never written to Postgres)
4. In-progress Postgres queries might be interrupted

With graceful shutdown:
1. App gets SIGTERM
2. Worker flushes remaining click buffer to Postgres
3. Worker closes (stops pulling new jobs)
4. App closes DB pool, Redis connection
5. Process exits cleanly

import { type Queue, QueueEvents } from "bullmq";
import { env } from "../config/env.js";
import { CLICK_QUEUE_NAME } from "./click-queue.js";

/**
 * Monitors the click queue for failed jobs that have exhausted all retries.
 * In production, this would alert your monitoring system (PagerDuty, Slack, etc.)
 */
export function monitorDeadLetters(): QueueEvents {
  const queueEvents = new QueueEvents(CLICK_QUEUE_NAME, {
    connection: {
      host: new URL(env.REDIS_URL).hostname,
      port: parseInt(new URL(env.REDIS_URL).port || "6379"),
      maxRetriesPerRequest: null,
    },
  });

  queueEvents.on("failed", ({ jobId, failedReason }) => {
    console.error(
      `[DeadLetter] Job ${jobId} permanently failed: ${failedReason}`
    );
    // In production, you'd send this to your alerting system:
    //   - Send Slack message
    //   - Increment a Prometheus counter
    //   - Log to a dead letter topic in Kafka (Week 5)
  });

  queueEvents.on("stalled", ({ jobId }) => {
    console.warn(`[DeadLetter] Job ${jobId} stalled`);
  });

  console.log("[DeadLetter] Monitoring for failed jobs...");

  return queueEvents;
}

A good cache hit rate for URL redirects: > 90%
