Q2: i thought SSL is needed for the web frontend domains didn't know backend alsoneeds them i understand TLS handshake which where both of them trust each other and pass data i guess this SSL certificatio which is needed to backend

A2: GOOD QUESTION — LET'S CLEAR THIS UP
-----------------------------------------
SSL/TLS is NOT about "frontend vs backend." It's about ENCRYPTING THE CONNECTION
between ANY two points that talk over a network.

WHERE SSL LIVES IN OUR SETUP:

    User (browser) ──── HTTPS (encrypted) ────→ Nginx ──── HTTP (plain) ────→ App:3000

    SSL is between the USER and NGINX. Not between Nginx and your app.
    This is called "SSL TERMINATION" — Nginx handles the encryption/decryption.
    Your app talks plain HTTP to Nginx (they're on the same Docker network, so it's safe).

WHY NOT JUST ON FRONTEND?
    Think about it: your frontend makes API calls to the backend.

    Without SSL on backend:
        Browser ──── HTTPS ────→ Frontend (React)
        Frontend ──── HTTP (plain text!) ────→ Backend API
        Anyone on the network can read: passwords, tokens, user data

    With SSL on backend (what we're doing):
        Browser ──── HTTPS ────→ Nginx (terminates SSL)
        Nginx ──── HTTP ────→ App (same private network, safe)

    In our case, Nginx IS the entry point for both frontend and backend.
    There's no separate frontend server — it's all one Nginx.

REAL PROD SCENARIOS:

    Scenario 1: API-only backend (like TinyScale)
        User → HTTPS → Nginx/Load Balancer (SSL termination) → HTTP → App
        SSL cert is on Nginx. App doesn't know about SSL.

    Scenario 2: Frontend + Backend separate
        User → HTTPS → CDN (serves React) → HTTPS → API Gateway → HTTP → Backend
        SSL cert is on the CDN and API Gateway. Backend doesn't handle SSL.

    Scenario 3: Microservices talking to each other (mTLS)
        Service A → HTTPS → Service B (both have certs, both verify each other)
        This is called mutual TLS — used in zero-trust networks (K8s service mesh).

BOTTOM LINE:
    - SSL protects data IN TRANSIT (on the network)
    - Anything that faces the internet needs SSL (frontend OR backend)
    - Internal services (behind a firewall/private network) often use plain HTTP
    - Our Nginx handles SSL so our app doesn't have to
    - The self-signed cert is for learning. Prod uses Let's Encrypt (free, trusted).

A3: THREE WAYS — you do NOT need to cancel the terminal or stop other containers.

    docker compose restart nginx

        docker compose down

A6:

FIRST — WHAT IS LOAD TESTING ACTUALLY DOING? (since you asked)
================================================================

Think of it like this. Your app is a restaurant.
    - 1 customer at a time? Easy. Chef handles it.
    - 10 customers? Fine, maybe slightly slower.
    - 100 customers walk in at the same time? Kitchen breaks.

k6 simulates those 100 customers. Here's exactly what happens:

    Time 0:00-0:30 → k6 gradually creates 0→50 "virtual users" (VUs)
    Time 0:30-1:30 → 50 VUs all running simultaneously
    Time 1:30-2:00 → ramps up to 100 VUs
    Time 2:00-3:00 → 100 VUs all running simultaneously
    Time 3:00-3:30 → ramps back down to 0

    Each VU runs the default function in a loop:
        1. Roll a dice:
           - 20% chance → POST /api/shorten (create a short URL)
           - 70% chance → GET /{code} (click a short URL, expect 302 redirect)
           - 10% chance → GET /api/stats/{code} (get analytics)
        2. Check: did I get the expected response? (201, 302, 200)
        3. If not → count as error
        4. Sleep 0.5-1.5 seconds (simulate human think time)
        5. Loop again

    With 100 VUs, each doing ~1 request/second → ~63 requests/second hitting your server.
    That's what http_reqs rate shows: 63.2/s.

A8: THE POINT IS NOT "SEND LOTS OF REQUESTS." THE POINT IS "FIND WHERE IT BREAKS."
-------------------------------------------------------------------------------------

Rate limiting is a PROTECTION feature. It works. You proved it.
Load testing has a DIFFERENT goal: find the performance limits of your actual
application logic — the code, the database, the memory.

Think of it this way:
    Rate limiting = the bouncer at the door (rejects too many people)
    Load testing = how many customers can the kitchen serve before food gets slow?

    You can't test the kitchen if the bouncer won't let anyone in.
    So for load testing, you temporarily remove the bouncer.

WHAT WE'RE ACTUALLY LOOKING FOR:

    1. DATABASE BOTTLENECK:
       At what point does Postgres slow down? 50 req/s? 200 req/s?
       When connections max out (pool size = 10), new queries wait in line.
       → This tells you: "I need connection pooling (PgBouncer) or Redis caching"

    2. CPU BOTTLENECK:
       Node.js is single-threaded. At what req/s does the CPU hit 100%?
       → This tells you: "I need multiple app instances (scaling)"

    3. MEMORY BOTTLENECK:
       Does memory grow under load? Any memory leaks?
       → This tells you: "I have a leak" or "I need more RAM"

    4. RESPONSE TIME DEGRADATION:
       At 10 req/s, response time = 5ms.
       At 100 req/s, response time = 50ms.
       At 500 req/s, response time = 2 seconds.
       → WHERE does it start getting slow? That's the bottleneck.

REAL EXAMPLE OF WHAT YOU SHOULD FIND:

    "Without caching, my redirect endpoint hits Postgres on every request.
     At 100 req/s, p(95) response time is 200ms.
     DB CPU is at 80%. That's my bottleneck.
     In Week 3, I'll add Redis caching — redirect checks Redis first,
     skips Postgres entirely. Then re-test and compare the numbers."

    THAT is what goes on your resume:
    "Identified database as bottleneck at 100 req/s via load testing.
     Added Redis caching, improved to 2000 req/s with p(95) < 10ms."

SO THE PLAN:
    1. Temporarily disable BOTH rate limiters (Nginx + app) ← just for testing
    2. Run the load test — let ALL requests through to your app
    3. Watch: CPU, memory, DB connections, response times
    4. Find the bottleneck (probably DB since you have no caching yet)
    5. Write down the numbers
    6. Re-enable rate limiters
    7. In Week 3: add Redis caching, re-test, compare numbers

    It's not about breaking things for fun. It's about MEASURING your system
    so you know WHAT to improve and can PROVE the improvement with numbers.

    GOOD CODE:  2000 req/s → CPU 80%, p(95) = 15ms

Same machine. 20x more throughput. The difference is HOW you use the resources.

WHAT CAN YOU ACTUALLY OPTIMIZE? (more than just "code")

    1. AVOID UNNECESSARY WORK (biggest win)
       Right now: every redirect → query Postgres → return URL
       With Redis: every redirect → check Redis (0.1ms) → return URL
       Postgres query = 5-20ms. Redis lookup = 0.1ms. 100x faster.
       Same machine, same CPU — just smarter about where you look.

    2. DO WORK LATER, NOT NOW (async processing)
       Right now: click happens → write to clicks table → THEN redirect user
       With queue: click happens → push to queue (0.1ms) → redirect immediately
       A worker processes the queue in the background.
       User gets redirected faster. Same machine.

    3. DO LESS WORK PER REQUEST (query optimization)
       SELECT * FROM urls WHERE short_code = 'abc'

       With index: Postgres jumps directly to the row (fast always)
       Same machine, same query, just an index.

    4. REUSE CONNECTIONS (pooling)
       Opening a new DB connection = 50ms overhead each time
       Reusing from a pool = 0ms overhead
       Same machine.

    5. SCALE HORIZONTALLY (multiple instances)
       Node.js = single-threaded = uses 1 CPU core
       Your laptop has 4+ cores. 3 cores sit idle.
       Run 3 app containers behind Nginx load balancer → 3x throughput.
       Same machine, just using ALL cores instead of one.

Q13: how do docker images get their names?

A13:
-----
Two ways images get named:

1. YOU NAME IT with -t when building manually:
       docker build -t tinyscale:latest .
                      ↑ you chose this name

2. DOCKER COMPOSE names it automatically:
       Format: {project-folder}_{service-name}

       Your docker-compose.yml has services: app, migrate, db
       Your folder is called "tinyscale" (or "TinyScale")

       So compose creates:
           tinyscale-app        ← from "app" service (built from your Dockerfile)
           tinyscale-migrate    ← from "migrate" service (same Dockerfile, target: builder)
           postgres:16-alpine   ← from "db" service (pre-built, keeps original name)
           nginx:alpine         ← from "nginx" service (pre-built, keeps original name)

       Run `docker images` and you'll see both:
           tinyscale-app       latest    150MB   ← compose auto-named
           tinyscale-migrate   latest    400MB   ← compose auto-named
           tinyscale           latest    150MB   ← your manual build from Step 5 (if still there)
           postgres            16-alpine 240MB   ← pulled from Docker Hub
           nginx               alpine    40MB    ← pulled from Docker Hub

NAMING CONVENTION:
    name:tag
    tinyscale:latest    ← name = tinyscale, tag = latest
    postgres:16-alpine  ← name = postgres, tag = 16-alpine
    node:22-alpine      ← name = node, tag = 22-alpine

    Tags are versions. "latest" is the default if you don't specify one.
    In prod you NEVER use "latest" — you pin a specific version so deploys are predictable.
